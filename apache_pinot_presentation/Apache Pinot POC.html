The Problem: Stale Data

  • 5-minute-old data = Lost business insights
  • Can't optimize in real-time
  • Miss fraud detection windows (decisions happen too late)
  • Dashboards don't reflect current reality

Traditional databases optimized for writes, not fast analytics.

Architecture Mismatch

MySQL, PostgreSQL, Redshift weren't built for real-time analytics.

  • Row-based storage – reads entire rows even for single columns
  • Transactional focus – optimized for INSERT/UPDATE/DELETE
  • Limited parallelism – queries compete for resources
  • Slow analytical queries – seconds to minutes on large data

The Solution: Apache Pinot

Sub-100ms queries on

Petabyte-Scale Data

Built for real-time analytics. Purpose-built.

Three Pillars of Pinot

1. Columnar Storage

Store by column, not row. Read only what you need. 90% less disk I/O.

2. Distributed

Scale horizontally. Add servers = more capacity. Query all in parallel.

3. Advanced Indexing

Inverted, sorted, range indexes. Know what to read before reading.

Schema Design (Your Setup)

Dimension Fields: userId, eventType (filter & group by)

Metric Fields: value (SUM, AVG, COUNT, MAX)

DateTime Fields: timestamp (time-based pruning)

✓ Simple, flat schema = Pinot's sweet spot
✗ Complex joins = Move upstream, denormalize data

Retention & Segments

  • 7-day retention – Keep recent data hot in REALTIME table
  • Hourly segments – 7 days × 24 hours = 168 segments
  • Parallel querying – Query all 168 simultaneously
  • Auto-cleanup – Data older than 7 days is deleted
  • + Batch layer – Optional: historical data from S3/Hadoop

Production: Hybrid REALTIME (7 days) + BATCH (2+ years) tables

Query Example 1: Events by Type

SELECT eventType, COUNT(*) FROM events WHERE timestamp > now() - 3600000 GROUP BY eventType ORDER BY COUNT(*) DESC

  • Timestamp index → only reads last hour segments
  • Columnar storage → reads only 2 columns (eventType, value)
  • Result: <50ms latency

Query Example 2: User Engagement

SELECT userId, AVG(value), COUNT(*) FROM events WHERE eventType = 'click' GROUP BY userId HAVING COUNT(*) > 10

  • Inverted index on eventType → skip non-click rows
  • Columnar → reads userId, value columns only
  • Scales to 1B+ events

Trade-offs to Know

  • No Complex Joins – Denormalize upstream (Kafka enrichment)
  • Fixed Schemas – Add columns easily, modify columns carefully
  • Immutable Data – No UPDATE/DELETE. Append-only design
  • Operational Complexity – More components than single-DB setup

These are design trade-offs for speed. Completely acceptable for analytics workloads.
