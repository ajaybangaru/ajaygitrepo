The Challenge

Why Traditional DBs Fail

It's not about hardware; it's about Architecture.

Transactional Focus

Optimized for ACID consistency. INSERT, UPDATE, DELETE take precedence over read speed.

Row-Based Storage

Reads the entire row (all columns) just to access one. Massive, unnecessary disk I/O.

Limited Parallelism

Single-node bottlenecks. Scaling up is expensive; scaling out is complex for analytics.

The "ETL" Tax

Data must be moved to a warehouse. This adds latency (minutes to hours).

The Architecture

The Three Pillars of Pinot

1. Columnar Storage

Store by column, not row. Read only what you need. 90% less disk I/O for analytical queries.

2. Distributed Design

Horizontal scale. Data is partitioned across servers. Parallel query execution on every segment.

3. Advanced Indexing

Inverted, Sorted, and Range indexes. Know what to read before touching the disk.

✓ Star-Tree ✓ Bloom Filters ✓ JSON Indexes

Proven at LinkedIn, Uber, and Walmart at Petabyte Scale

Key Parameters

REALTIME Type

Enables continuous ingestion from streaming sources like Kafka.

Auto-purges data older than 7 days to manage storage costs.

Low-Level Consumer

Directly maps Kafka partitions to Pinot segments for maximum parallelism.

Live Analytics

Querying the Stream

1. Real-Time Distribution

How many events occurred per type in the last hour?

SELECT 
  eventType, COUNT(*)
FROM events
WHERE timestamp > now() - 3600000
GROUP BY eventType
ORDER BY COUNT(*) DESC

Pinot Optimization: Uses the timestamp index to prune segments, then uses the dictionary-encoded eventType for instant grouping.

2. User-Centric Metrics

What is the engagement level for active users?

SELECT 
  userId, AVG(value), COUNT(*)
FROM events
WHERE eventType = 'click'
GROUP BY userId
HAVING COUNT(*) > 10

Pinot Optimization: Columnar storage reads only userId and value. Inverted index on eventType skips non-click rows entirely.

Live Analytics

Advanced Time-Series

Dynamic Time-Bucketing

Hourly aggregation for trend analysis (Dashboard ready):

SELECT 
  FROM_UNIXTIME(CAST(timestamp / 1000 AS BIGINT), 'yyyy-MM-dd HH:00:00') as hour_bucket,
  eventType,
  COUNT(*) as event_count,
  AVG(value) as avg_value
FROM events
WHERE timestamp > now() - 86400000
GROUP BY hour_bucket, eventType
ORDER BY hour_bucket DESC

Key Technique

Dividing timestamp by 1000 converts milliseconds to seconds for standard SQL date functions.

Latency Target

Even with complex bucketing, Pinot targets < 150ms for this query on billions of rows.
